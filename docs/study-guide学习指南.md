## 技术栈详解

### ⭐⭐⭐ Spring Security + JWT 认证授权

**核心概念：**

1. **Spring Security 过滤器链 (Filter Chain)**
   - 每个请求经过一系列 Filter，每个 Filter 负责一项安全检查
   - `SecurityContextPersistenceFilter` → `UsernamePasswordAuthenticationFilter` → `ExceptionTranslationFilter` → `FilterSecurityInterceptor`
   - 我们添加了自定义 `JwtAuthenticationFilter`，在 `UsernamePasswordAuthenticationFilter` 之前执行

2. **JWT (JSON Web Token) 结构：**
   - **Header：** 算法类型（HS256）、Token 类型（JWT）
   - **Payload：** 声明（Claims），存放用户信息、过期时间等
   - **Signature：** 对 Header 和 Payload 的签名，防止篡改
   - 格式：`xxxxx.yyyyy.zzzzz`

3. **JWT vs Session：**

| 对比维度 | JWT | Session |
|---------|-----|---------|
| 状态 | 无状态 | 有状态 |
| 存储位置 | 客户端 | 服务端内存/Redis |
| 扩展性 | 天然支持分布式 | 需要 Session 共享 |
| 安全性 | Token 泄露风险 | 相对更安全 |
| 主动失效 | 困难（需要黑名单） | 容易（删除 Session） |

4. **BCrypt 密码加密：**
   - 每次加密自动生成随机盐值（Salt）
   - 同一密码每次加密结果不同
   - 可配置加密强度（cost factor）
   - 相比 MD5/SHA 更安全

### ⭐⭐ Redis 缓存策略

1. **缓存使用场景：**
   - 商品列表缓存（减少数据库查询）
   - 用户 Session / Token 黑名单
   - 热点数据缓存（热销商品排行榜）
   - 计数器（商品访问量、点赞数）

2. **缓存问题及解决方案：**
   - **缓存穿透：** 查询不存在的数据 → 布隆过滤器 / 缓存空值
   - **缓存雪崩：** 大量缓存同时过期 → 过期时间加随机值 / 互斥锁
   - **缓存击穿：** 热点 key 过期 → 互斥锁 / 逻辑过期

3. **Redis 序列化：**
   - StringRedisSerializer：Key 用 String 序列化
   - Jackson2JsonRedisSerializer：Value 用 JSON 序列化，便于跨语言

### ⭐⭐ RabbitMQ 消息队列

1. **应用场景：**
   - **异步处理：** 订单创建后发送通知、更新库存
   - **削峰填谷：** 秒杀/大促时请求先入队列，后端按能力处理
   - **应用解耦：** 订单服务 → 消息队列 → 库存服务/物流服务

2. **核心概念：**
   - **Queue：** 消息队列，存储消息
   - **Exchange：** 交换机，决定消息路由到哪个队列
   - **Binding：** 绑定关系，连接 Exchange 和 Queue
   - **Routing Key：** 路由键，Exchange 根据它分发消息

3. **交换机类型：**
   - **Direct：** 精确匹配 Routing Key
   - **Topic：** 通配符匹配 Routing Key
   - **Fanout：** 广播到所有绑定队列
   - **Headers：** 根据消息头属性匹配

### ⭐⭐ MyBatis-Plus

相比 MyBatis 的优势：
- 无需手写基本 CRUD SQL
- Lambda 表达式条件构造器
- 分页插件一键集成
- 自动填充（createTime 等公共字段）
- 乐观锁插件、逻辑删除

### ⭐⭐ 推荐系统原理

1. **协同过滤 (Collaborative Filtering)**
   - **User-based：** 找到与目标用户相似的用户，推荐他们喜欢的商品
   - **Item-based：** 找到与目标商品相似的商品，推荐给用户
   - **优点：** 不需要商品特征，完全基于用户行为
   - **缺点：** 冷启动问题、计算量大

2. **基于内容的推荐 (Content-based)**
   - 根据商品特征（分类、标签、价格区间）推荐相似商品
   - **优点：** 没有冷启动问题
   - **缺点：** 特征工程依赖人工

3. **混合推荐 (Hybrid)**
   - 综合多种推荐算法，取长补短
   - 加权融合 / 切换融合 / 级联融合

4. **冷启动问题：**
   - **用户冷启动：** 新用户无历史数据 → 热销推荐 / 随机推荐 / 问卷调查
   - **商品冷启动：** 新商品无交互数据 → 基于内容推荐 / 人工标注
   - **系统冷启动：** 全新平台 → 导入外部数据 / 编辑推荐

### ⭐ 微服务架构概念

- **服务拆分：** 按业务功能拆分为独立服务
- **服务发现：** Nacos / Eureka
- **配置中心：** Nacos / Apollo
- **网关路由：** Spring Cloud Gateway
- **负载均衡：** Ribbon / LoadBalancer
- **熔断降级：** Sentinel / Hystrix
- **链路追踪：** SkyWalking / Zipkin

## 面试考点

### 1. Spring Security JWT 工作原理

```
流程图：
1. 客户端发送登录请求 → POST /api/auth/login
2. 服务端验证用户名密码 → 生成 JWT Token → 返回给客户端
3. 客户端保存 Token → 后续请求携带 Authorization: Bearer <token>
4. JwtAuthenticationFilter 拦截请求 → 提取 Token → 验证签名 → 解析 Claims
5. 验证通过 → 设置 SecurityContext → 执行 Controller 方法
6. 验证失败 → 返回 401 Unauthorized
```

**追问1：Token 过期了怎么办？**
> 双 Token 机制：Access Token（短时效，如 30 分钟）+ Refresh Token（长时效，如 7 天）
> 客户端发现 Access Token 过期 → 使用 Refresh Token 获取新的 Access Token

**追问2：如何主动让 JWT 失效？**
> 维护一个 Token 黑名单（Redis），每次请求检查 Token 是否在黑名单中

### 2. Redis 缓存策略

**常见问题：**
- 如何保证缓存与数据库一致性？（先更新数据库，再删除缓存 / 延迟双删）
- 缓存穿透如何解决？（布隆过滤器 + 缓存空值）
- 热点 key 问题？（本地缓存 + 分布式缓存多层架构）

### 3. MyBatis-Plus 性能优化

- 大表分页优化（使用覆盖索引 / 游标分页）
- 批量操作（saveBatch 合并 SQL）
- 懒加载（延迟加载关联数据）

### 4. 推荐系统评估指标

- **准确率 (Precision)：** 推荐商品中用户喜欢的比例
- **召回率 (Recall)：** 用户喜欢的商品中被推荐的比例
- **覆盖率 (Coverage)：** 推荐商品占总商品的比例
- **多样性 (Diversity)：** 推荐结果的差异化程度
- **新颖性 (Novelty)：** 推荐用户不知道的商品

### 5. 系统设计扩展

**如果日活用户达到 100 万，如何扩展？**
1. **水平扩展：** 多副本部署后端服务 + 负载均衡
2. **缓存分层：** 本地缓存（Caffeine）→ 分布式缓存（Redis）→ 数据库
3. **读写分离：** MySQL 主从架构，读流量走从库
4. **分库分表：** 按用户 ID 分片（ShardingSphere）
5. **异步化：** 非核心逻辑走消息队列（订单通知、统计等）
6. **CDN：** 商品图片静态资源走 CDN
7. **限流熔断：** 接口限流（Sentinel）+ 熔断降级
